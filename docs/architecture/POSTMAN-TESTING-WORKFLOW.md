# ProPhoto Sandbox Postman Workflow

## Purpose

This collection provides a one-click workflow for loading fresh sandbox seed data into Postman and running smoke tests against the ProPhoto sandbox API.

It is designed to work with Laravel seed output that generates machine-readable Postman variables such as:

- `PROPHOTO_API_BASE_URL`
- `SESSION_ID`
- `PROPHOTO_BEARER_TOKEN`
- `STUDIO_BEARER_TOKEN`
- `CLIENT_BEARER_TOKEN`
- `SUBJECT_BEARER_TOKEN`
- `STUDIO_EMAIL`
- `CLIENT_EMAIL`
- `SUBJECT_EMAIL`

## Postman workflow

1. `sandbox.json` fresh **runtime** values generated by the sandbox seeder
2. `postman-requests.json` source of truth for request definitions
3. `postman-collection.json` generated collection **imported** into Postman

## Recommended Structure

### 00 - Setup
- **Load Sandbox Context**: Load sandbox context from seeder output.

### 01 - Smoke Tests
- **Check Session Progress**: Verify session status and progress.
- **Check Preview Status**: Verify asset preview generation.
- **Confirm Session**: Verify session confirmation idempotency.

### 02 - Workflow Actions
- Future workflow or non-smoke operational requests.

## How This Workflow Works

1. Your Laravel sandbox seeder generates fresh runtime data.
2. That runtime data is copied from `sandbox.json` and loaded into the active Postman environment via the **Load Sandbox Context** request.
3. The smoke-test requests run using only environment variables.
4. Validation scripts confirm the API is behaving correctly.
5. **Rule**: No hardcoded bearer tokens or per-seed IDs should exist in requests.

## Setup Instructions

1. **Refresh Sandbox**: Seed or refresh the sandbox in Laravel.
2. **Copy Data**: Copy the generated JSON from `sandbox.json`.
3. **Open Postman**: Go to the **Load Sandbox Context** request.
4. **Paste JSON**: Paste the JSON into the raw request body.
5. **Send**: Send the request.
6. **Result**: The script will parse the payload and save any matching variables into the active environment.

### Supported Payload Shapes for Setup

The setup request supports multiple JSON formats to allow the seeder output to evolve without breaking the Postman workflow:
- Flat JSON
- Nested `postman` object
- Nested `data.postman` object

### Expected Behavior of "Load Sandbox Context"

The setup request script should:
- Parse valid JSON from the request body.
- Look for Postman variables in supported locations.
- Save matching keys into the active environment.
- Ignore missing optional values.
- Fail clearly if the JSON is invalid or empty.

## Smoke Test Workflow

**Recommended Order:**
1. **Load Sandbox Context**
2. **Check Session Progress**
3. **Check Preview Status**
4. **Confirm Session**

> **Note**: *Confirm Session* is designed to be safe in both first-run and repeat-run scenarios. If the session has already been completed, the API may return a response indicating it was already processed rather than creating duplicate assets.

## Smoke Test Expectations

### Check Session Progress
- **Purpose**: Verifies the seeded session exists and reports expected progress/status fields.
- **Validation**:
    - HTTP response is successful (200 OK).
    - Response body is valid JSON.
    - Session identifier is present.
    - Session status fields match the API contract.
    - Progress/count fields are present and valid.
    - Completion flags are interpreted using actual API field names.

### Check Preview Status
- **Purpose**: Verifies preview generation state and asset visibility for the seeded session.
- **Validation**:
    - HTTP response is successful.
    - Response body is valid JSON.
    - Session exists.
    - Preview or thumbnail data shape is valid.
    - Expected arrays and counts are present.
    - Completion-related values are consistent.

### Confirm Session
- **Purpose**: Verifies session confirmation behavior.
- **Validation** (Two valid scenarios):
    1. First-run confirmation succeeded.
    2. Repeat-run/idempotent confirmation returned an "already-processed" outcome.
- **Rule**: This request should not fail simply because the session was already completed if that is expected business behavior.

## Collection Request Rules

All requests in this collection must follow these rules:
- **Environment Driven**: Use variables (`{{variable}}`) for base URLs, IDs, emails, and tokens.
- **No Hardcoding**: Never hardcode bearer tokens or session UUIDs that come from seed output.
- **Dynamic Auth**: Keep auth headers variable-driven.
- **Variable Bodies**: Keep request bodies variable-driven where practical.
- **Clear Assertions**: Add test scripts with clear pass/fail assertions.
- **Contract Accuracy**: Write tests against actual response fields, not guessed field names.

## Source Files and Responsibilities

| File | Responsibility |
| :--- | :--- |
| `sandbox.json` | **Runtime Seed Output**: Contains fresh per-seed values; changes every reseed; not committed. |
| `postman-requests.json` | **Source of Truth**: Human-readable flat request metadata used for PR review; committed to source. |
| `postman-collection.json` | **Import Artifact**: Generated Postman collection compiled from metadata and supporting logic. |

## Adding New Requests

1. **Define**: Add a new entry to `postman-requests.json`.
2. **Parameterize**: Use variables instead of hardcoded values.
3. **Categorize**: Put the request in the correct folder:
    - `00 - Setup`: Bootstrap/setup work.
    - `01 - Smoke Tests`: Pass/fail verification.
    - `02 - Workflow Actions`: Operational steps.
4. **Validate**: Add scripts checking status code, JSON validity, and key contract fields.
5. **Regenerate**: Rebuild `postman-collection.json` through the seeder/build step.
6. **Import**: Re-import the collection into Postman.

## Authoring Guidance

When creating requests:
- Prefer descriptive names.
- Make required variables obvious.
- Avoid hidden assumptions about session state.
- Write tests that tolerate valid idempotent behavior.
- Treat Postman as a consumer of generated artifacts, not the primary editing surface.
